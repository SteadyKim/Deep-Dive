# DB 파티셔닝

## 개요

파티셔닝은 하나의 큰 테이블이나 인덱스를 논리적으로는 하나로 유지하면서, 물리적으로는 여러 파티션으로 나누어 저장하고 관리하는 기법입니다.

대용량 데이터를 효율적으로 조회하고, 특정 구간 데이터를 빠르게 보관 또는 삭제하며, 유지보수 작업의 범위를 줄이기 위해 사용합니다.

## 파티셔닝 종류

### 레인지 파티셔닝

숫자나 날짜처럼 연속된 값의 범위를 기준으로 분할합니다.  
예를 들어 매출 테이블을 판매 일자 기준으로 월마다 나눌 수 있습니다.

- 대량의 과거 데이터를 이력별로 관리하기 좋습니다.
- 특정 기간 데이터를 일괄 삭제하거나 보관할 때 효율적입니다.
- 날짜 기반 조회가 많은 시스템에서 자주 사용됩니다.

### 리스트 파티셔닝

지역 코드나 부서 번호처럼 불연속적인 특정 값의 목록을 기준으로 분할합니다.

- 데이터 분포가 특정 코드값에 따라 명확히 구분되는 경우에 적합합니다.
- 지정되지 않은 값이 들어올 경우를 대비해 나머지 값을 처리하는 파티션을 둘 수 있습니다.
- 조직, 지역, 서비스 유형처럼 범주 기준 분리에 유리합니다.

### 해시 파티셔닝

파티션 키 값을 해시 함수에 대입하여 나온 결과에 따라 데이터를 균등하게 배분합니다.

- 특정 구간에 데이터가 몰리는 현상을 줄일 수 있습니다.
- 입출력 부하를 비교적 고르게 분산시키는 데 적합합니다.
- 정렬 순서보다는 부하 분산과 병렬 처리 효율 향상이 목적입니다.

### 복합 파티셔닝

여러 파티셔닝 방식을 조합하는 방식입니다.  
예를 들어 날짜로 먼저 레인지 분할을 한 뒤, 각 날짜 파티션 내부에서 다시 지역 코드로 리스트 분할을 할 수 있습니다.

- 관리 효율과 성능 최적화를 동시에 노릴 수 있습니다.
- 조회 패턴과 운영 요구사항이 복잡한 대용량 시스템에서 자주 사용됩니다.
- 단일 방식보다 설계와 운영 난이도가 높습니다.

## 인덱스 아키텍처

파티션 테이블에서 인덱스는 물리적 저장 구조에 따라 크게 로컬 인덱스와 글로벌 인덱스로 나뉩니다.

### 로컬 인덱스

테이블 파티션과 인덱스 파티션이 1:1로 대응되는 구조입니다.

- 특정 날짜 파티션을 삭제하면 해당 날짜의 인덱스 조각도 함께 제거됩니다.
- 파티션 단위 관리가 쉬워 운영 부담이 적습니다.
- 대용량 배치 작업 후 인덱스 영향 범위가 해당 파티션으로 제한됩니다.
- 파티션 단위 유지보수와 함께 사용할 때 가장 자연스럽습니다.

### 글로벌 인덱스

테이블의 파티션 구조와 별개로 인덱스가 독자적인 구조를 가지거나, 아예 파티션되지 않은 형태입니다.

- 여러 파티션에 걸친 데이터를 하나의 인덱스 기준으로 식별할 수 있습니다.
- 기본키나 유니크 제약 조건을 전체 범위에서 보장해야 할 때 사용됩니다.
- 특정 파티션을 삭제하거나 분할하는 작업이 발생하면 인덱스 전체에 영향이 갈 수 있습니다.
- 유지보수 시 인덱스 unusable 상태나 재구성 필요성을 함께 고려해야 합니다.

## 정리

- 레인지 파티셔닝은 기간 단위 관리에 적합합니다.
- 리스트 파티셔닝은 범주형 값 분리에 적합합니다.
- 해시 파티셔닝은 데이터 분산과 부하 균형에 적합합니다.
- 복합 파티셔닝은 성능과 관리 효율을 함께 고려한 방식입니다.
- 로컬 인덱스는 관리 편의성이 높습니다.
- 글로벌 인덱스는 전체 범위 제약 조건 관리에 유리하지만 유지보수 부담이 큽니다.
