# SGA 경유 방식과 Direct Path 방식의 입출력

## 개요

Oracle 계열 아키텍처를 기준으로 보면, 데이터 I/O는 크게 두 흐름으로 이해할 수 있습니다.

- `SGA` 안의 버퍼 캐시를 거치는 방식
- 버퍼 캐시를 우회하는 `Direct Path` 방식

둘 다 결국 물리 디스크 I/O를 수행한다는 점은 같습니다.  
그런데 애플리케이션이 체감하는 속도는 꽤 다르게 나타날 수 있습니다.

핵심은 단순히 "디스크를 쓰느냐"가 아니라, **누가 기다리는지**, **어떤 패턴으로 기록하는지**, **I/O를 얼마나 묶어서 처리할 수 있는지**에 있습니다.

## 먼저 알아둘 개념

- `SGA(System Global Area)`는 여러 세션이 공유하는 메모리 영역입니다.
- `Buffer Cache`는 `SGA` 안에서 데이터 블록을 임시로 보관하는 캐시 영역입니다.
- `Direct Path`는 버퍼 캐시를 거치지 않고 프로세스가 직접 읽거나 쓰는 방식입니다.
- `LOB Direct Path`는 대용량 `CLOB`나 `BLOB` 처리에서 버퍼 캐시보다 직접 I/O가 더 적합하다고 판단될 때 나타날 수 있는 방식입니다.

## 한눈에 비교

| 구분 | SGA 경유 방식 | LOB Direct Path 방식 |
| --- | --- | --- |
| 경로 | 세션 → 버퍼 캐시(SGA) → 백그라운드 프로세스가 디스크 기록 | 세션/서버 프로세스가 디스크에 직접 I/O |
| 세션 대기 | 상대적으로 짧음 | 상대적으로 김 |
| I/O 패턴 | 버퍼링, 묶음 처리에 유리 | 개별 처리, 직접 기록 비중이 큼 |
| 재사용성 | 높음 | 낮음 |
| 적합한 상황 | 반복 접근, 캐시 재사용이 중요한 작업 | 대용량 LOB 처리, 캐시 오염을 줄여야 하는 작업 |

## 왜 둘 다 디스크 I/O를 하는데 체감 속도가 다를까

겉으로 보면 두 방식 모두 결국 디스크에 기록합니다.  
하지만 애플리케이션 관점에서 느껴지는 속도 차이는, 디스크 기록 자체보다 **사용자 세션이 어떤 I/O를 직접 기다리는지**에서 갈립니다.

## 1. 사용자 세션의 대기 시점

### SGA 경유 방식

SGA를 경유하는 방식에서는 변경 내용이 먼저 메모리 영역인 버퍼 캐시에 반영됩니다.  
실제 데이터 파일에 더티 버퍼를 내려쓰는 작업은 보통 `DBWR` 같은 백그라운드 프로세스가 나중에 처리합니다.

즉, 사용자 세션 입장에서는 **데이터 파일 쓰기 자체를 항상 정면으로 기다리는 구조가 아닙니다.**

물론 커밋 시점에는 장애 복구를 위해 `redo log` 기록을 기다릴 수 있지만, 적어도 "데이터 파일 블록 전체를 즉시 직접 쓰는 부담"은 줄어듭니다.

### LOB Direct Path 방식

반면 `LOB Direct Path`는 서버 프로세스가 `LOB` 세그먼트에 직접 I/O를 수행하는 성격이 강합니다.  
이 경우 사용자 세션은 실제 물리 쓰기가 끝날 때까지 더 직접적으로 대기하게 됩니다.

즉, 같은 디스크 I/O라도 **세션이 기다리는 I/O의 종류와 시점이 다르기 때문에 체감 속도 차이**가 생깁니다.

## 2. 물리적 I/O 패턴

### SGA 경유 방식은 상대적으로 순차적이고 예측 가능한 쓰기에 유리

SGA 경유 방식에서도 `redo log`는 즉시 디스크에 기록될 수 있습니다.  
하지만 리두 로그는 보통 파일 끝에 이어 쓰는 `Sequential I/O` 성격이 강하기 때문에, 저장장치 입장에서는 처리 효율이 높습니다.

즉, 사용자는 디스크를 기다리더라도 **상대적으로 빠른 형태의 쓰기**를 기다리는 경우가 많습니다.

### LOB Direct Path 방식은 랜덤 I/O 부담이 커지기 쉬움

대용량 HTML 같은 `CLOB` 데이터를 `LOB` 세그먼트에 직접 기록할 때는 상황이 달라집니다.  
비어 있는 청크를 찾아가며 쓰거나, 여러 위치에 흩어져 기록하는 `Random I/O` 성격이 강해질 수 있습니다.

이 경우 저장장치는 단순히 "쓰기 한 번"을 하는 것이 아니라, **어디에 쓸지 찾고, 위치를 옮기고, 여러 군데에 나눠 기록하는 비용**까지 감당해야 합니다.

그래서 같은 물리 I/O여도 체감 속도는 훨씬 느리게 나타날 수 있습니다.

## 3. 디스크 기록의 효율성

### SGA 경유 방식은 일괄 처리에 유리

버퍼 캐시에 쌓인 더티 버퍼는 `DBWR`가 모아서 디스크에 내려쓸 수 있습니다.  
이 과정에서 데이터베이스와 운영체제는 물리적 위치가 가까운 블록을 묶거나, 더 효율적인 순서로 처리하면서 I/O 비용을 줄일 수 있습니다.

즉, 메모리에 한 번 모아뒀다가 쓰기 때문에 **쓰기 최적화 여지**가 큽니다.

### LOB Direct Path 방식은 세션 단위 개별 처리 성격이 강함

`LOB Direct Path`는 대량 데이터가 들어오는 즉시, 단일 세션이 직접 공간을 확보하고 기록해야 하는 경우가 많습니다.  
이 방식은 버퍼 캐시에 쌓아두고 묶어서 쓰는 최적화의 이점을 충분히 누리기 어렵습니다.

결국 같은 양을 기록하더라도, **묶어서 효율적으로 쓰는 경우와 그때그때 개별로 쓰는 경우의 차이**가 성능 차이로 이어집니다.

## 정리하면

`LOB Direct Path`가 느리게 체감되는 이유는 "디스크를 쓰기 때문"이 아니라, 아래 세 가지가 동시에 겹치기 쉽기 때문입니다.

1. 사용자 세션이 직접 물리 쓰기를 더 오래 기다립니다.
2. 순차 쓰기보다 랜덤 쓰기 성격이 강해집니다.
3. 백그라운드 일괄 처리의 이점을 얻기 어렵습니다.

즉, 두 방식은 모두 디스크 I/O를 수행하지만, **세션이 기다리는 방식과 디스크가 처리하는 형태가 다르기 때문에 애플리케이션 체감 성능이 극명하게 갈릴 수 있습니다.**
