# 쉐어드 풀과 파싱 메커니즘

## 개요

오라클 아키텍처에서 `Shared Pool`은 SQL 실행 비용을 줄이기 위해 사용하는 핵심 공유 메모리 영역입니다.  
같은 SQL이 반복 실행될 때 매번 처음부터 해석하고 최적화하지 않도록, SQL 문장과 실행 계획, 메타데이터를 메모리에 보관합니다.

즉, 쉐어드 풀의 핵심 목적은 **쿼리 실행에 필요한 연산 자원 소모를 줄이고, 이미 계산한 결과를 최대한 재사용하는 것**입니다.

## 쉐어드 풀이 필요한 이유

SQL 한 문장이 실행되기 전에는 단순히 "문자열"인 상태에 가깝습니다.  
데이터베이스는 이 문자열을 실제로 실행하기 위해 아래 작업을 거쳐야 합니다.

- 문법이 올바른지 확인
- 테이블, 뷰, 컬럼이 실제로 존재하는지 확인
- 사용자에게 해당 객체에 대한 접근 권한이 있는지 확인
- 어떤 인덱스를 쓸지, 어떤 조인 순서가 유리한지 계산
- 최종 실행 계획을 생성

이 과정은 생각보다 비용이 큽니다.  
그래서 오라클은 한 번 분석한 SQL과 실행 계획을 쉐어드 풀에 저장하고, 다음 실행부터는 최대한 재사용하려고 합니다.

## 쉐어드 풀의 핵심 구성 요소

### 1. 라이브러리 캐시

`Library Cache`는 사용자가 실행한 SQL 문장, 컴파일된 프로시저 코드, 그리고 옵티마이저가 생성한 실행 계획을 저장하는 영역입니다.

이 영역의 핵심 역할은 **SQL 재사용**입니다.

- 입력된 SQL 문장을 기준으로 캐시를 탐색합니다.
- 동일하거나 재사용 가능한 SQL이 이미 존재하면 기존 실행 계획을 활용합니다.
- 없으면 새로 실행 계획을 만들고 다시 캐시에 등록합니다.

즉, 라이브러리 캐시는 "이 SQL을 전에 본 적이 있는가?"를 판단하는 곳이라고 볼 수 있습니다.

### 2. 데이터 딕셔너리 캐시

`Data Dictionary Cache`는 테이블 구조, 컬럼 타입, 인덱스 정보, 사용자 권한 같은 메타데이터를 저장하는 영역입니다.

이 캐시는 데이터 자체를 저장하는 곳이 아니라, **데이터베이스를 이해하고 제어하기 위한 기준 정보**를 보관하는 곳입니다.

예를 들어 아래 정보가 이 영역에서 활용됩니다.

- 테이블과 뷰의 존재 여부
- 컬럼 정의
- 인덱스 유무
- 사용자 권한
- 오브젝트 관련 메타정보

다른 캐시들이 주로 블록 단위 데이터를 다루는 것과 달리, 이 캐시는 로우 단위 성격의 메타데이터를 캐싱하기 때문에 `Row Cache`라고도 부릅니다.

## 쿼리 실행 시 쉐어드 풀 내부 처리 흐름

사용자가 새로운 SQL을 실행하면 서버 프로세스는 먼저 쉐어드 풀을 중심으로 아래 흐름을 수행합니다.

### 1. 구문 및 의미 분석

먼저 SQL 문법이 올바른지 확인합니다.  
그다음 `Data Dictionary Cache`를 조회해 다음 내용을 검사합니다.

- SQL에 명시된 테이블이나 뷰가 실제로 존재하는지
- 참조한 컬럼이 올바른지
- 실행한 사용자에게 해당 객체에 대한 권한이 있는지

즉, 이 단계는 SQL이 "실행 가능한 문장인지" 검증하는 과정입니다.

### 2. 라이브러리 캐시 탐색

기본 검증이 끝난 SQL 문장은 내부 해시 함수에 전달됩니다.  
오라클은 이 결과를 바탕으로 `Library Cache`를 탐색해, 이미 같은 SQL 또는 재사용 가능한 실행 계획이 존재하는지 확인합니다.

핵심은 매번 캐시를 처음부터 전부 뒤지는 것이 아니라, **해시 기반으로 빠르게 후보를 찾는다**는 점입니다.

### 3. 소프트 파싱

캐시에서 재사용 가능한 SQL과 실행 계획을 찾았다면 `Soft Parse`가 발생합니다.

이 경우에는 아래 비용을 대부분 줄일 수 있습니다.

- 복잡한 최적화 연산
- 실행 계획 재생성
- 불필요한 CPU 소모

즉, 이미 만들어진 실행 계획을 꺼내 바로 실행 엔진으로 넘기기 때문에 가장 이상적인 처리 경로에 가깝습니다.

### 4. 하드 파싱

라이브러리 캐시에서 재사용 가능한 실행 계획을 찾지 못하면 `Hard Parse`가 발생합니다.

이때는 옵티마이저가 개입해 가능한 실행 경로를 본격적으로 계산합니다.

- 사용할 수 있는 인덱스 확인
- 조인 순서 조합
- 조인 방식 선택
- 예상 비용 계산
- 최적 실행 계획 생성

완성된 실행 계획은 이후 재사용할 수 있도록 다시 `Library Cache`에 등록됩니다.

## 소프트 파싱과 하드 파싱 비교

| 구분 | 소프트 파싱 | 하드 파싱 |
| --- | --- | --- |
| 실행 계획 재사용 | 가능 | 불가능 |
| 옵티마이저 개입 | 거의 없음 | 적극적으로 개입 |
| CPU 사용량 | 적음 | 큼 |
| 응답 속도 | 빠름 | 상대적으로 느림 |
| 시스템 부담 | 낮음 | 높음 |

실무에서는 SQL이 최대한 소프트 파싱되도록 만드는 것이 매우 중요합니다.  
동일한 의미의 SQL을 매번 다른 문자열 형태로 보내면, 재사용이 어려워져 하드 파싱이 자주 발생할 수 있기 때문입니다.

## 흐름으로 보면

```text
사용자 SQL 요청
→ 문법 검사
→ 데이터 딕셔너리 캐시로 객체/권한 확인
→ 라이브러리 캐시 탐색
→ 있으면 소프트 파싱
→ 없으면 하드 파싱 후 실행 계획 생성
→ 실행 계획을 캐시에 저장하고 실행
```

## 정리

- `Shared Pool`은 SQL 실행 비용을 줄이기 위한 핵심 공유 메모리 영역입니다.
- `Library Cache`는 SQL 문장과 실행 계획을 저장해 재사용을 돕습니다.
- `Data Dictionary Cache`는 테이블 구조, 권한, 인덱스 같은 메타데이터를 캐싱합니다.
- `Soft Parse`는 기존 실행 계획을 재사용하는 빠른 경로입니다.
- `Hard Parse`는 옵티마이저가 새 실행 계획을 만드는 무거운 경로입니다.

결국 쉐어드 풀은 오라클이 SQL을 더 적은 비용으로 반복 실행할 수 있도록 만드는 핵심 메모리 구조라고 볼 수 있습니다.
