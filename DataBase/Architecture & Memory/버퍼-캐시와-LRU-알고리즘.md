# 버퍼 캐시와 LRU 알고리즘

## 개요

데이터베이스 성능에서 가장 큰 병목 중 하나는 디스크 I/O입니다.  
디스크는 메모리보다 훨씬 느리기 때문에, 같은 블록을 반복해서 디스크에서 읽기 시작하면 응답 시간이 급격히 늘어납니다.

이 문제를 줄이기 위해 데이터베이스는 자주 접근하는 데이터 블록을 메모리 영역인 `Buffer Cache`에 보관합니다.

## 버퍼 캐시의 목적

버퍼 캐시의 핵심 목적은 **디스크 접근을 줄이고, 더 빠르게 데이터를 제공하는 것**입니다.

데이터를 요청하면 시스템은 먼저 버퍼 캐시를 확인합니다.

- `Cache Hit`: 요청한 블록이 이미 메모리에 있는 상태
- `Cache Miss`: 요청한 블록이 메모리에 없어 디스크에서 다시 읽어와야 하는 상태

흐름은 아래처럼 정리할 수 있습니다.

```text
사용자 요청
→ 버퍼 캐시 확인
→ 있으면 메모리에서 즉시 반환 (Cache Hit)
→ 없으면 디스크에서 읽어와 캐시에 적재 후 반환 (Cache Miss)
```

즉, 버퍼 캐시는 디스크 I/O를 완전히 없애는 장치는 아니지만, 반복 접근이 많은 환경에서 성능 차이를 크게 만드는 핵심 메모리 구조입니다.

## LRU 알고리즘이 필요한 이유

버퍼 캐시는 메모리 공간이 제한되어 있기 때문에, 새로운 블록을 계속 적재하다 보면 언젠가는 기존 블록을 밀어내야 합니다.

이때 어떤 블록을 제거할지 판단하는 기준으로 사용되는 대표적인 방식이 `LRU(Least Recently Used)` 알고리즘입니다.

`LRU`는 이름 그대로 **가장 오랫동안 사용되지 않은 블록을 우선 제거하는 방식**입니다.  
최근에 접근한 블록은 다시 사용될 가능성이 높고, 오래 접근하지 않은 블록은 다시 사용될 가능성이 상대적으로 낮다는 전제를 활용합니다.

## 단계별 동작 메커니즘

### 1. 새로운 블록이 들어오면 MRU 쪽에 배치

디스크에서 새로운 블록을 읽어 버퍼 캐시에 적재하면, 해당 블록은 가장 최근에 사용된 영역인 `MRU(Most Recently Used)` 쪽에 배치됩니다.

### 2. 시간이 지나면 오래된 블록이 뒤로 밀림

새로운 블록이 계속 들어오면 기존 블록들은 상대적으로 뒤쪽으로 밀립니다.  
오랫동안 다시 사용되지 않은 블록은 점차 `LRU` 쪽으로 이동합니다.

### 3. 다시 접근한 블록은 앞으로 이동

이미 캐시에 있는 블록에 다시 접근하면, 그 블록은 다시 최근 사용 블록으로 판단되어 `MRU` 쪽으로 이동합니다.  
즉, 자주 참조되는 블록일수록 캐시에 더 오래 남게 됩니다.

### 4. 공간이 부족하면 LRU 쪽 블록부터 제거

메모리 여유가 부족해지면, 가장 오래 사용되지 않은 `LRU` 쪽 블록부터 제거됩니다.  
그렇게 확보한 공간에 새 블록을 적재합니다.

## 한눈에 정리

| 항목 | 의미 |
| --- | --- |
| 버퍼 캐시 | 자주 사용하는 데이터 블록을 메모리에 보관하는 공간 |
| Cache Hit | 요청한 블록이 캐시에 있어 바로 읽을 수 있는 상태 |
| Cache Miss | 요청한 블록이 캐시에 없어 디스크에서 읽어와야 하는 상태 |
| MRU | 가장 최근에 사용된 블록이 위치하는 쪽 |
| LRU | 가장 오래 사용되지 않은 블록이 밀려나는 쪽 |

## 마무리

- 버퍼 캐시는 디스크 I/O 지연을 줄이기 위한 핵심 메모리 구조입니다.
- `Cache Hit` 비율이 높을수록 데이터베이스는 더 빠르게 동작합니다.
- `LRU`는 제한된 메모리 안에서 어떤 블록을 유지하고 어떤 블록을 제거할지 결정하는 기준입니다.

결국 버퍼 캐시와 LRU 알고리즘은, 제한된 메모리로 최대한 많은 재사용 효과를 끌어내기 위한 데이터베이스의 기본 전략이라고 볼 수 있습니다.
